# eSociety "Residents" App - Architecture & Workflow Explanation

The `residents` app is designed to manage the physical structure of the society and the people living within it. It uses a "Service Layer" pattern, which is a senior-level Django best practice. This means we separate our business logic (database queries) from our routing logic.

Below is an explanation of the core files and the most important workflow:

---

## 1. models.py (The Database Structure)
This file defines two groups of tables in our database:

**Group A: Physical Infrastructure**
- `Society`: The root object. Represents the entire housing complex.
- `Building`: Belongs to a Society (e.g., "Block A").
- `Flat`: Belongs to a Building. This is the actual apartment unit.

**Group B: People & Assets**
- `ResidentProfile`: The core of this app. It links the `core.User` (which handles login/passwords) to a specific `Flat`. It also tracks their move-in date.
- `FamilyMember`: Linked to a `ResidentProfile`. Allows a resident to register their spouse, children, etc.
- `Vehicle`: Linked to a `ResidentProfile`. Tracks cars and bikes for parking allocation.

---

## 2. services.py (The Brain/Business Logic)
In typical beginner Django, all database queries are jammed into `views.py`. As a senior developer, we extract this into `services.py`. This file contains pure Python functions that handle the heavy lifting of talking to the database.

- **Why?** It makes the code reusable. If we want to add a family member from a mobile app API later, we just call `services.add_family_member()` instead of rewriting the logic.
- **Key Functions:**
  - `create_resident_profile()`: Safely creates a profile for a user without duplicating it.
  - `assign_flat()`: Lets an admin link a resident to a specific apartment.
  - `get_resident_dashboard_data()`: Aggregates all data (profile, flat, family, vehicles) into one dictionary so the dashboard view can easily render it.

---

## 3. views.py (The Traffic Cop)
Because we moved the heavy lifting to `services.py`, our views are incredibly clean and simple. Their only job is to:
1. Check permissions (Is the user logged in? Are they an admin?).
2. Grab data from the request (URL parameters, POST form data).
3. Call the appropriate function in `services.py`.
4. Return an HTML template or redirect the user.

- `resident_directory_view`: Used by Admins. Calls `services.list_residents()` to show everyone in the society.
- `add_vehicle_view`: Used by Residents. Takes form data and calls `services.add_vehicle()`.

---

## Example Workflow: A Resident Registers a New Car

Let's walk through exactly what happens when a user clicks "Submit" to add a new car to their profile:

1. **The Starting Point (Browser):**
   The resident is logged in and goes to `http://127.0.0.1:8000/residents/vehicle/add/`. They fill out the HTML form with their car's license plate ("MH12AB1234") and select "Car" from the dropdown. They click "Submit".

2. **The Router (`urls.py`):**
   Django receives the POST request. `urls.py` sees the `/vehicle/add/` path and routes the traffic to the `add_vehicle_view` function in `views.py`.

3. **The Traffic Cop (`views.py`):**
   The `add_vehicle_view` function takes over:
   - The `@login_required` decorator confirms the user is logged in.
   - It sees this is a `POST` request, so it extracts the data:
     `number = request.POST.get('number')` (which is "MH12AB1234")
   - It fetches the logged-in user's `ResidentProfile` object.
   - It delegates the actual creation to the service layer by calling:
     `services.add_vehicle(profile.id, number, vehicle_type, parking_slot)`

4. **The Brain (`services.py`):**
   The `add_vehicle()` function runs:
   - It fetches the `ResidentProfile` using the provided ID safely.
   - It executes the database query: `Vehicle.objects.create(...)`.
   - The new vehicle is permanently saved in the database.
   - It returns the newly created Vehicle object back to the view.

5. **The Finish Line (`views.py` again):**
   The view function finishes its execution by redirecting the user's browser back to their main dashboard:
   `return redirect('resident_dashboard_data')`
   
6. **The Result:**
   The browser reloads the dashboard, and the resident now sees their newly registered car listed on the page!
